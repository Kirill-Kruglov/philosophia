## 3. Where the reviews differ — strictest resolution, named

1. **Recovery after an invalid run.** Opus AM-1 would classify a clean,
   mechanically evidenced resource stop as the already-signed A6
   process-re-execution category (re-executable without consuming the
   floor-amendment budget). Sol M5/edit 5 forbids any invalidity
   subtype from authorizing a rerun. **Sol is stricter and governs:**
   an invalid run authorizes nothing automatically; any future
   mechanically justified resource recovery requires a **new explicit
   Kirill-signed process/resource decision plus bounded review, with
   the learner rule unchanged**. It would not be a third floor
   amendment, but this amendment does not pre-authorize it. Opus's
   substantive concern — that a mere timeout must not permanently and
   *silently* block Level 1 — is still met: the block is loud, named,
   and reversible only by an explicit signed decision.
2. **Residual Level 0 anchor scope.** Opus would allow "full-batch
   viability at this scale" as genuine provenance for adopting the
   regime; Sol restricts Level 0 to "an engineering precedent that
   full-batch AdamW ran on this platform for a different locked task,"
   supplying no choice among repairs. **Sol is stricter and governs**
   (and the update-count fact Opus surfaced — Level 1's 2,000 sits
   below Level 0's 5,200–7,700 — is retained as a reason the anchor
   cannot support the old framing).
3. **"Parameter-free."** Opus accepts the term for the rule's syntax;
   Sol requires "no new numeric hyperparameter" plus loud naming as a
   capacity/optimization-policy change. **Sol is stricter and governs.**
4. **Opus Q3 pre-check.** Opus advises it; Sol is silent. Conservative
   route: **not folded into this amendment**; eligible only as its own
   separately authorized, capped, non-outcome check (§1 note).

No Opus edit was dropped by these resolutions; each strict Sol reading
contains the corresponding Opus repair as a subset or leaves it intact.
