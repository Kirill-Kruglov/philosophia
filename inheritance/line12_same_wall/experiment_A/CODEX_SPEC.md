# Спецификация для независимой реализации (передаётся в НОВУЮ сессию, без истории)

Ты пишешь программы, которые исследуют неизвестный мир через оракул.
Никакого другого контекста у этой задачи нет и не должно быть.

## Мир (что известно)

Существует скрытое множество состояний и два действия: `R` и `L`. Слово —
конечная строка над алфавитом `{R, L}` — переводит стартовое состояние в
некоторое конечное. Мир может быть устроен по-разному: часто `L`
действует как обратное к `R` и орбита замыкается в цикл некоторого
периода `m`; но мир может быть и не таким (хвосты, липкие состояния,
несогласованные действия), а оракул может изредка ошибаться. Ничего
больше про мир не гарантируется.

## Интерфейс (единственный канал к миру)

```python
oracle(u: str, v: str) -> bool
# True, если слова u и v из стартового состояния приводят в ОДНО состояние
```

Бюджет: не более `cap_calls` вызовов оракула на один запуск. Случайность:
только собственный `random.Random(seed)` из переданного `seed`; никаких
глобальных `random.*`.

## Что нужно вернуть

Каждая программа имеет сигнатуру:

```python
def find_order(oracle, cap_calls: int, seed: int):
    ...
    return ('VAL', m, confident)   # найден период m; confident: bool
    return ('AB',)                 # мир противоречит допущениям — отказ
    return ('TO',)                 # бюджет исчерпан без вывода
```

Семантика: `('VAL', m, True)` — уверенное утверждение «период орбиты
равен m»; `('VAL', m, False)` — то же с низкой уверенностью; `('AB',)` —
осознанный отказ (обнаружена несовместимость с моделью периодического
мира); `('TO',)` — не успел.

## Задание

1. **`find_order_main`** — твой основной искатель периода. Дизайн
   полностью твой: любые слова (не только степени одного действия),
   любая стратегия проб, любая внутренняя модель. Требования: работать
   при `cap_calls` порядка нескольких сотен; деградировать честно
   (AB/TO, а не уверенная ложь) на мирах, нарушающих периодичность; быть
   устойчивым к редким ошибкам оракула настолько, насколько сочтёшь
   нужным (задокументируй выбор).
2. **`find_order_ref1`, `find_order_ref2`, `find_order_ref3`** — три
   дополнительных искателя, НАРОЧНО построенных на взаимно различных
   механизмах (различных и от основного). Цель — разнообразие подходов к
   одной задаче. Кратко задокументируй, чем механизмы различаются.
3. **Мемо дизайна** (в комментариях файла): для каждой функции — идея,
   внутренние компоненты, как используется случайность, какие допущения
   о мире она делает и когда абстинирует.

## Ограничения

- Один файл, чистый Python stdlib, без импортов кроме `random`, `math`.
- Функции детерминированы при фиксированных (oracle, cap_calls, seed).
- Не запрашивай дополнительный контекст — всё, что известно, написано
  здесь. Если чего-то не хватает, сделай выбор сам и задокументируй его.
